## コンテキスト

月末 (5/29) の engine review を起点に、`fx_protocol` の UGH v2 engine
内部値をコード照合まで深掘りし、6 月着手の改善計画 (`docs/engine_review_2026_05_planning.md`)
を PR #104 として作成した。本書は `Yuu6798/semantic-ci-code/docs/*_planning.md`
の style convention (status banner / 定量現状 / non-goals / 各 fix の
target.yaml / Phase 化 / Acceptance criteria 二段 / Open Questions /
Appendix) に準拠。

序盤は週次ログ分析 (5/11-5/15、5/18-5/22、5/25-5/29) を通じて UGH 予測の
構造的問題を可視化 (range_hit 100% の実体が 15/15、dominant_state の
knife-edge 遷移、FLAT 0 件、fire 1 日のみ)。その後、コード + 4 サンプル
検算で「conviction の符号反転バグ疑い」を実は仕様通りと判定し直すなどの
裏取り pass を経て、最終的に P0/P1/P2 ラベル付きの 750+ 行 planning doc に
落とし込んだ。

PR #104 は Codex P2 review を 13 ラウンド (合計 22 review threads) 受け、
すべて対応してマージ。

## 設計判断

- **計画書スタイルは semantic-ci-code 準拠**: 既存の repo に planning doc
  convention がなかったため、`Yuu6798/semantic-ci-code` の `*_planning.md`
  を style template として採用。status banner、定量現状、non-goals、各 fix の
  target.yaml 雛形、Phase 化、Acceptance Criteria 二段 (ship / knowledge)、
  Open Questions、Appendix (data sources / code citation index)。
- **engine_version の bump cadence**: Phase 1 v2 維持 → Phase 2 v2.1
  (dominant_state) → Phase 3 v2.2 (FLAT) → Phase 3 v2.3 (range)。理由は
  `stratify_observations_by_versions` の auto-detect が `theory_version`
  しか分離せず、bump しないと pre/post が同じ報告バケットに混在するため。
  → 後に round 12 で **Phase 2 内で auto-detect を `engine_version`
  にも拡張** する systemic fix に変更 (per-caller plumbing より durable)。
- **FLAT epsilon の動的閾値**: 静的 5 bp 案 → round 8 で「5/26 サンプル
  (+11〜+14 bp) で `|11| > 5` のため UP のまま」と指摘され、`max(ratio *
  trailing, floor_bp)` に切り替え (ratio=0.5 / floor=5.0)。**全変種一律
  FLAT を要求せず**、variant 探索趣旨で direction disagreement を許容する
  acceptance に softening。
- **fire dual criterion**: round 4 で「event-day 軸と variant-record 軸の
  両方を criterion」と決定。1/15 → 3+/15 (event 日) かつ 3/60 → 10+/60
  (sample × variant)。**両方満たさないと loophole** (同じ fire-event 日に
  variant が増えるだけ) を許容してしまうことを発見。
- **`ugh_v2_ensemble` 行の導入** (round 10): `weekly_strategy_metrics.csv`
  に synthetic 行を追加し、`range_hit_count` / `range_hit_rate` は
  ensemble 行のみが持つ (4 つの variant 行は空文字列)。`is_ugh_kind` も
  `ugh_v2_ensemble` を認識するよう拡張して、`compute_monthly_baseline_comparisons`
  が pollute されないようにした。
- **conviction の二重役割は spec 改訂のみ**: 検算で仕様通り動作と判明 →
  P2 として spec docstring 改訂で対応、コード変更なし。

## 成功パターン

- **コード照合 + 検算による裏取り**: weekly 分析だけだと「バグかも」で
  止まる issue を、4 サンプル (5/8/26/27/29) の implied evidence_confidence
  逆算 (0.92-0.99 で安定) や、`_build_range_from_baseline_context` の
  variant ProjectionConfig 未渡し確認など、code:line + データ両方で
  根拠を固められた。
- **PR description ベースの review loop**: Codex P2 review に対して
  「コード再確認 → ドキュメント修正 → commit hash 付きで reply」という
  ループが安定。毎 round 1-2 件のレビューを 10 分以内に turn around。
- **systemic fix の選択**: round 12 で「per-caller `engine_version_filter`
  plumbing」と「`stratify_observations_by_versions` の auto-detect 拡張」の
  2 択を提示され、後者を選んだことで以降の bump phase (3, B) で target.yaml
  に `report_window` を再掲する必要がなくなった。
- **Codex がチェイン的に発見**: 13 rounds の中で round 5 (state metrics
  per-variant) → round 9 (4.4 fire validation の dual criterion 反映) →
  round 13 (Open Question #4 の caller plumbing 削除) のように、ひとつの
  修正が次の不整合を表面化させるパターン。誠実に毎 round 対応した結果、
  最後に "Chef's kiss. No major issues" でクローズ。

  含んでいたが、13 rounds の Codex review で網羅的に洗い出された。

  §12 を一緒に更新するのを忘れ、結果として「caller plumbing が必要」という

  重要。

## メモ

- **13 rounds は本リポジトリ最多**。fx_protocol Phase 1-3b (5/8 セッション)
  での最多が 2-3 rounds だったので、planning doc は実装より review surface
  が大きいことを実感。1 つの fix が周辺の整合性問題を波及させる構造。
- **`ToolSearch select:mcp__github__*` のパターン**が定着。MCP server が
  intermittent に切断/再接続するため、毎回 ToolSearch で reload する習慣。
- **Codex review に対して reply を必ず付ける**運用が定着。コード変更
  ハッシュ (`<commit>`) と修正箇所の前後コンテキストを quote すると、後で
  thread を見返したときに「何を直したか」がすぐ分かる。
- **`forecast_batch_id` dedupe** は実装時に最も間違えやすい箇所。
  `labeled_observations.csv` の `forecast_batch_id` が 4 variants で
  完全共有されているため、`set()` 経由で 1 行に collapse する必要がある
  (`_compute_metrics_for_rows` は row ごとに `range_hit` をカウントする)。
- **`is_ugh_kind` 拡張パターン**: monthly review の baseline 比較から
  ensemble を除外するために、`UGH_V2_ENSEMBLE_KIND` 定数 + `is_ugh_kind`
  の文字列分岐に追加。raw observation filter は ensemble が CSV に出ない
  ため double-count しない。
- 本セッションが `.claude/memory/` convention の 2 件目利用。前回 (5/8) が
  meta-introduction で、今回が初の実運用。
