## 設計判断

- **メール通知**: SMTP (`dawidd6/action-send-mail`) を選択。GitHub Issue ではなく直接
  メール、トリガーは「新コミット push 時のみ」(idempotent_skip ランは黙る)。本文は当初
  `run_summary.json` だったが「内容が読めない」フィードバックを受けて `daily_report.md`
  に変更、さらに Claude Sonnet 4.6 で生成する日本語 TL;DR を冒頭に追加。
- **anthropic install の non-fatal 化** (Codex P2 review): メイン setup から外して
  通知ステップ手前に `pip install anthropic || true` で配置。データ生成自体をブロック
  しないよう徹底。
- **`__all__` の取り扱い** (Codex P2 review): `from observability import *` の
  既存 public 名が消えないよう `__all__` には全 public 名 (13 件) を列挙。
  以後 PR #96, #103 でも同パターンを踏襲。
- **Phase 3 を 4 サブフェーズに分割**: 元のスケッチが「byte-identical な抽出 (低リスク)」
  と「load-bearing なスキーマ統一 (高リスク・ROI 低)」を一括にしていたのを是正。3a/3b
  だけ実装、3c (loader bridge) と 3d (metric row schema unification) は明示的に defer。
- **テストカバレッジ整理は白紙化**: 25 件の spike で「Pydantic 重複は 16% しか当たら
  ず、削減見込み 4-7% (当初 7-11%)」と判明したため ROI 不足と判断。

## メモ

- Codex review への反論は thread に投稿するパターンが定着 (PR #92, #93)。
  「コード変更せず説明で済ます」判断のときは reasoning を残すと未来の自分が助かる。
- semantic-ci の `effects_unchanged` violation は refactor で必ず出るので、PR description で
  毎回「expected」と明記する運用が定着。Phase 1.5 / 3a / 3b では純粋ヘルパーだったため
  4/4 satisfied、Phase 1 / 2 / 3 で 3/4 + expected。`primary_kind: feature` に
  変えれば抑止できるが意味的に refactor が正しいので維持。
- `__all__` 拡張パターンは PR #93 review thread で確立、以降 PR #96, #103 で踏襲。
  「狭い `__all__` を ruff F401 抑制目的で書くと star-import を破壊する」は再発しがちな
  ので、新規 `__all__` 追加時は必ず既存 public 名を全列挙する。
- ToolSearch 経由の deferred tool ロード (mcp__github__update_pull_request 等) は何度
  も使った。直接ロードできる tool が初期からは見えないことがあるので、必要になった
  ら ToolSearch する習慣がついた。
- 本セッションがそのまま `.claude/memory/` convention の最初の利用者になる予定だった
  が、PR #103 merge 後に main からの wrap-up となるため `2026-05-08.md` (本ファイル)
  の生成は session 末尾でしか起こらない。次回セッションは `_index.md` + 本ファイルを
  読んで context を引き継げる想定。
